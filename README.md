## Домашнее задание 8: Мониторинг ML-системы

### 1. Ключевые SLO и дерево метрик

На основе требований бизнеса, разработчиков, ML-команды и DevOps было построено дерево метрик. Основные целевые уровни SLO:

- **Бизнес (выручка)**
    - ARPU ≥ 5₽
    - Конверсия (CTR) ≥ 3.5%
    - LTV ≥ 150₽
- **DevOps (надёжность/ресурсы)**
    - Доступность (SLA) 99.9%
    - Latency p99 ≤ 200 мс
    - Error Rate 5xx < 0.1%
    - Utilisation CPU ≤ 70%
- **ML-команда (качество модели)**
    - Offline: NDCG@10 ≥ 0.65
    - Online: CTR ≥ 3.5%
    - PSI (дрейф признаков) < 0.1
    - Prediction Latency (модель) ≤ 50 мс
- **Разработчики (бэкенд/инфра)**
    - RPS на сервис = 10k
    - Throughput бэкенда ≥ 8k RPS
    - Оптимальный размер батча 32–64
    - Ошибки Docker-in-Docker < 1%

> Дерево метрик визуализировано в ноутбуке с помощью `networkx`.

### 2. Настройка мониторинга (Prometheus + Grafana + MLflow)

Выбрана метрика: **p95 latency** модели.

#### Шаги настройки:
1. Запущен `service.py` с эндпоинтами `/predict` и `/metrics` (гистограмма `request_latency_seconds`).
2. Поднят Prometheus через Docker с конфигом `prometheus.yaml`, скрапящим метрики с сервиса.
3. В Grafana импортирован дашборд `ml_system_monitoring.json` с панелью:
   - **p95 Request Latency** (формула: `histogram_quantile(0.95, sum(rate(request_latency_seconds_bucket[5m])) by (le))`)
   - Установлен алерт: p95 > 1s в течение 2 минут.

#### Результат:
- Метрики собираются в Prometheus, визуализированы в Grafana.
- При превышении порога срабатывает алерт с сообщением:
  > p95 latency exceeded 1s for 2 minutes. Investigate model inference.

### 3. Дрейф данных / концепции и деградация модели

Смоделированы три сценария с использованием библиотеки **Evidently**:

1. **Дрейф данных (Data Drift)**  
   Сдвиг распределения признака `score` (нормальное смещение +2, σ=0.5).  
   Результат: `DataDriftPreset` показал высокий PSI (>0.1), качество модели упало с 0.85 до 0.67.

2. **Дрейф концепции (Concept Drift)**  
   Изменено правило формирования таргета:  
   раньше `target = (score > 0.2)`, теперь `target = (score <= 0.2)`.  
   Точность модели упала до 0.23.

3. **Деградация модели + дрейф**  
   Комплексное искажение всех признаков.  
   Точность упала с 0.85 до 0.41.  
   Отчёт Evidently подтвердил сильный дрейф по всем метрикам качества данных.

### 4. Data Quality Ops (DQOps)

#### Шаги реализации:
1. Установлен DQOps через `pip install dqops`.
2. Поднят сервер DQOps (порт 8080), подключён туннель через `tuna`.
3. В PostgreSQL создана тестовая таблица `employees`.
4. Импортирована структура таблицы в DQOps.
5. Выполнено изменение схемы: удалена колонка `email`.
6. Повторно запущена проверка качества данных.

#### Инцидент:
Скриншот в ноутбуке показывает обнаружение отсутствия обязательной колонки `email`.  
SQL, вызвавший инцидент:

```sql
SELECT
    COUNT(*) as total_records,
    SUM(CASE WHEN email IS NULL THEN 1 ELSE 0 END) as null_emails,
    ROUND(SUM(CASE WHEN email IS NULL THEN 1 ELSE 0 END) * 100.0 / COUNT(*), 2) as null_percentage
FROM employees;
-- Результат: ERROR: column "email" does not exist
```

→ **Инцидент зафиксирован, качество данных нарушено.**

### 5. Архитектура ML-системы (Virtual Product Placement)

Разработана схема системы на базе **Kappa-архитектуры** с потоковой обработкой видео (Redis Streams + FastStream).  
Диаграмма включает:
